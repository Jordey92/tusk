# Tusk

[![CI](https://github.com/jordey92/tusk/actions/workflows/ci.yml/badge.svg?branch=main)](https://github.com/jordey92/tusk/actions/workflows/ci.yml)

Tusk keeps PostgreSQL schema changes explicit: you write paired SQL files, and
Tusk validates, orders, locks, runs, and tracks them.

- Plain SQL instead of a migration DSL
- One transaction per migration
- Advisory locking to prevent concurrent migration runs
- CLI and programmatic APIs for Node.js and Bun

## Install

Tusk requires Node.js 18+ or Bun 1.3.8+, PostgreSQL 13+, and one supported
driver. Most projects can start with `pg`:

```bash
npm install @bydey/tusk pg
```

Using Bun:

```bash
bun add @bydey/tusk pg
```

For postgres.js, install `postgres` instead of `pg`.
Programmatic imports use ESM; the CLI also works in CommonJS projects.

## First Migration

Set your database URL in `.env`:

```dotenv
DATABASE_URL=postgresql://user:password@localhost:5432/app
```

Create the migration files:

```bash
npx tusk init
npx tusk create create_users
```

Edit the generated pair:

```sql
-- migrations/<timestamp>_create_users.up.sql
CREATE TABLE users (
  id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  email TEXT NOT NULL UNIQUE
);
```

```sql
-- migrations/<timestamp>_create_users.down.sql
DROP TABLE users;
```

Validate, inspect, and apply:

```bash
npx tusk validate
npx tusk doctor
npx tusk up --dry-run
npx tusk up
npx tusk status
```

Bun users can replace `npx` with `bunx`.

## Commands

| Command | Purpose |
| --- | --- |
| `tusk init` | Create the migrations directory |
| `tusk create <name>` | Create paired `.up.sql` and `.down.sql` files |
| `tusk validate` | Check migration files without applying them |
| `tusk doctor` | Run a read-only project and database preflight |
| `tusk up --dry-run` | Print the pending migration plan |
| `tusk up` | Apply pending migrations |
| `tusk down [count]` | Roll back the latest migration, or an explicit count |
| `tusk down --all` | Roll back every applied migration |
| `tusk status` | Show applied and pending migrations |

Add `--json` for machine-readable output; its stable fields are documented in
[JSON output contracts](./docs/json-contracts.md). Use `tusk validate --db` to
compare files with database history without changing migration state.

## How Tusk Protects Migration History

Migration files use the format `<timestamp>_<name>.up.sql` and
`<timestamp>_<name>.down.sql`. Tusk records applied up migrations and their
checksums in `_migrations`; see the [metadata table contract](./docs/metadata-table.md).

Each file and its metadata update run in one transaction. Earlier files remain
committed if a later migration fails. A session advisory lock prevents two
runners from changing the same migration history concurrently.

`tusk down` rolls back one migration by default. Wider rollback requires an
explicit count or `--all`. Tusk resolves every required down file before it
starts the rollback batch.

## Existing Databases

Use `tusk init --from-db` only when adopting a schema that already exists. It
creates and records a protected baseline instead of replaying that schema.
Test this against a disposable copy first; baseline generation intentionally
rejects PostgreSQL features it cannot reproduce safely.

See [Existing database adoption](./docs/existing-databases.md) before using it.

## Programmatic Use

```ts
import { Pool } from "pg";
import { runUp } from "@bydey/tusk";
import { createPgAdapter } from "@bydey/tusk/pg";

const pool = new Pool({ connectionString: process.env.DATABASE_URL });

try {
  await runUp(createPgAdapter(pool), "./migrations");
} finally {
  await pool.end();
}
```

Use `@bydey/tusk/postgres` with postgres.js. Implementing another client
requires the [custom adapter contract](./docs/custom-adapters.md). Tusk also
provides a verified Elysia plugin at `@bydey/tusk/elysia`, which requires Node.js
20+ or Bun 1.3.8+.

## Configuration

| Variable | Default | Purpose |
| --- | --- | --- |
| `DATABASE_URL` | - | PostgreSQL connection string |
| `MIGRATIONS_PATH` | `./migrations` | Migration directory |
| `TUSK_DRIVER` | Automatic; `pg` preferred | Driver override: `pg` or `postgres` |
| `TUSK_STATEMENT_TIMEOUT_MS` | `300000` | Statement timeout inside each migration transaction; `0` keeps the database default |
| `LOG_LEVEL` | `warn` | `debug`, `info`, `warn`, or `error` |

Individual `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_USER`, and `DB_PASSWORD`
variables can be used instead of `DATABASE_URL`.

See [Transactions and timeouts](./docs/transactions.md) before changing the
statement timeout or running unusual PostgreSQL DDL.
