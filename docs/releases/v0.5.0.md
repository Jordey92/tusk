# v0.5.0

v0.5.0 is the doctor release. It adds a single read-only command for checking whether a Tusk project and PostgreSQL target are ready before a human, script, or AI agent runs a mutating migration command.

## What's New

### Doctor preflight

`tusk doctor` runs the important project and database checks in one place:

```bash
tusk doctor
tusk doctor --json
```

It checks:

- the running Tusk version
- whether the migrations directory exists
- migration filename, pair, SQL, duplicate timestamp, and transaction-control validation
- database configuration
- database connectivity
- PostgreSQL engine and version support
- migration metadata readability
- checksum metadata availability
- checksum drift between local files and executed records
- migration status readability
- advisory lock acquisition and release

The command is intentionally read-only. It is meant to be safe to run before `tusk init`, `tusk up`, or `tusk down`.

### Agent-friendly JSON output

`tusk doctor --json` returns a structured report with:

- `ok`
- `summary`
- `environment`
- `database`
- `checks`

Agents and scripts can branch on check IDs such as `database.engine`, `database.drift`, and `database.advisoryLock` instead of parsing human text.

### Safer PostgreSQL target detection

Doctor now positively identifies supported PostgreSQL targets instead of assuming every PostgreSQL-like server is safe.

- PostgreSQL is supported.
- Aurora PostgreSQL is supported when detected.
- Amazon Redshift fails with a clear explanation.
- Unknown PostgreSQL-compatible engines fail closed instead of being treated as normal PostgreSQL.

This keeps the command conservative for hosted databases and compatibility layers.

### Better setup documentation

The new [doctor guide](https://github.com/Jordey92/tusk/blob/v0.5.0/docs/doctor.md) explains how to run the command, what each check means, how JSON output is shaped, and why Redshift is rejected.

The [agent workflow guide](https://github.com/Jordey92/tusk/blob/v0.5.0/docs/agents.md) now starts with `doctor --json` as the recommended preflight before validation, dry-run planning, or applying migrations.

## Upgrade Notes

- This release adds a new command and does not change existing migration behavior.
- `tusk doctor` exits `0` when no checks fail and `1` when one or more checks fail.
- Warnings do not fail the command. They indicate degraded or review-worthy state.
- Redshift and unknown PostgreSQL-compatible engines are reported as unsupported by doctor.
