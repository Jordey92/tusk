# v0.3.0

v0.3.0 is the agentic migrations release. It makes Tusk easier and safer for humans, scripts, and AI agents to inspect migration state, validate work, preview SQL, and operate on existing databases with clearer structured output.

## What's New

### Agent-friendly CLI output

- `--json` support was expanded across the CLI so automation can parse command results directly.
- JSON success and error responses use stable envelopes with `ok`, `command`, and structured payload/error fields.
- Normal informational logs are kept out of stdout when JSON output is requested.

Useful commands:

```bash
tusk create add_widgets --json
tusk validate --json
tusk validate --db --json
tusk up --dry-run --json
tusk up --json
tusk status --json
```

### Dry-run migration planning

Tusk can now return the exact ordered SQL plan before applying changes.

```bash
tusk up --dry-run
tusk up --dry-run --json
tusk down 1 --dry-run
tusk down 1 --dry-run --json
```

This gives agents and reviewers a safer inspect-before-apply workflow.

### Safer validation and checksum checks

- `tusk validate` checks migration filenames, pairs, SQL content, duplicate timestamps, and transaction-control statements.
- `tusk validate --db` performs read-only database checks against `_migrations`.
- Applied migrations are tracked with checksums so drift between local migration files and database history can be detected.
- Legacy `_migrations` tables without checksum metadata continue to work for read-only checks.

### Existing database takeover

`tusk init` now creates a baseline migration and records it as already applied in `_migrations`.

That means an existing database can be adopted without immediately re-running the generated baseline SQL:

```bash
tusk init --json
tusk status --json
tusk up --dry-run --json
```

New migrations can then continue through the normal create, validate, dry-run, and apply loop.

### MCP server for AI agents

v0.3.0 adds the `tusk-mcp` stdio server with tools for agent workflows:

- `tusk_validate`: validate migration files, with optional read-only database checks.
- `tusk_status`: return migration status for a configured PostgreSQL database.
- `tusk_plan_up`: return the ordered up-migration dry-run plan.
- `tusk_plan_down`: return the ordered down-migration dry-run plan.
- `tusk_create_migration`: create paired `.up.sql` and `.down.sql` migration files.

Start it with:

```bash
tusk-mcp
```

### Agent workflow documentation

A new agent workflow guide documents the recommended safe loop:

1. create a migration pair
2. edit both migration files
3. validate locally
4. run read-only database validation
5. review the dry-run plan
6. apply the migration
7. confirm status
8. run the relevant tests

See `docs/agents.md` for the full workflow and production safety notes.

### Repeatable quality tooling

v0.3.0 adds quality scripts for confidence checks around future migration work:

```bash
bun run quality:crap
bun run quality:mutation
bun run quality:agentic
```

These cover CRAP analysis, mutation testing, and the combined agentic quality gate.

## Upgrade Notes

- PostgreSQL migration state may gain checksum metadata during mutating migration operations.
- `tusk init` is now intentionally database-mutating because it records the generated baseline as applied.
- Use dry-run and JSON output before applying changes in automation or agent workflows.
