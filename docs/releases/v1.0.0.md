# Tusk 1.0.0

Tusk v1 establishes a small, explicit compatibility contract for SQL-first
PostgreSQL migrations across Node.js and Bun.

## Compatibility

- Node.js 18+, Bun 1.3.8+, and PostgreSQL 13+
- ESM programmatic API and CLI support from CommonJS projects
- driver-neutral root package with optional `/pg`, `/postgres`, and `/elysia`
  entrypoints
- Elysia runtime support on Node.js 20+ and Bun 1.3.8+

## Safety Changes From v0.5

- `up`, `down`, and dry-run plans enforce migration validation before execution
- migration files require strict numeric timestamps, paired directions,
  executable SQL, safe filenames, and no transaction-control statements
- migration creation is path-contained, exclusive, length-bounded, and safe
  under concurrent calls
- advisory locks retain one physical session, reject same-adapter concurrency,
  verify unlock success, and fail on transaction-pooled endpoints
- adopted baselines cannot be selected by ordinary rollback; the explicit
  `--allow-baseline-rollback` override is required even for its destructive plan
- `status` is read-only and treats missing metadata as empty state
- existing-database adoption fails before writing files or metadata when the
  schema contains unsupported PostgreSQL features
- migration batches use a configurable statement timeout (five minutes by default)

These changes may reject inputs or endpoints that v0.x accepted accidentally.
They close correctness and data-loss risks rather than preserving unsafe
pre-v1 behavior.

## Public Contracts

v1 freezes the documented package entrypoints, exported TypeScript types and
typed errors, CLI commands and option meanings, JSON envelopes, migration
filename/checksum ordering, `_migrations` semantics, and rollback defaults.
Deep `dist/` imports remain internal.

## Release Evidence

Every v1 release candidate passes unit, integration, package-consumer,
minimum-runtime, platform, CRAP, mutation, dependency-audit, and PostgreSQL 13
and 18 lanes. The publication workflow also requires guarded hosted evidence
from Neon or Supabase and RDS against the same immutable tarball, with Aurora
evidence when available. The GitHub release carries its checksum, CycloneDX
SBOM, and redacted hosted-provider evidence.

See the [compatibility policy](../compatibility.md), [transaction model](../transactions.md),
and [existing-database adoption guide](../existing-databases.md) before upgrading.
